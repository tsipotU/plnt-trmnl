**Date:** 2026-05-07
**Status:** Approved (brainstormed with Emiel 2026-05-07; no design open questions)

**Source:** In-app feedback round 2026-05-07 (rows #39–#49; #27 noise, kept open by request). Triaged into 11 GitHub issues — 8 Wave 19, 3 v1.1 backlog (#209 profile-pic variant, #210 add-flow rework, #211 acknowledgement pattern).

## Goal

Close the eight credibility + polish issues that surfaced in the second post-public-flip dog-food round. Three clusters:

1. **Bugs that erode trust** (#202 watering desync, #204 splash branching, #208 mother-plant picker) — visible inconsistencies between what the UI says and what's true.
2. **/add surface UX** (#205 placeholders, #206 pot-size labels, #207 form validation) — same form, batchable in one rebuild loop.
3. **Visual polish** (#201 Header wordmark + emoji, #203 emoji sweep round 2) — finish what Wave 17's emoji direction (#167) started.

Wave 19 is **polish + bugfix**, not architectural. No new primitives, no new endpoints (with one possible exception — #207 may need a small validation-error helper).

## Why these together

All eight came from a single dog-food session. Bundling lets the local-rebuild loop amortize across the whole wave (rebuild Docker → verify all eight surfaces → push) instead of paying that cost per issue. The clusters also share files:

- `/add` cluster (#205 + #206 + #207 + #208) all touch `AddPlant.tsx` and its helpers
- Emoji sweep (#203) touches plant-detail history + `EnrichmentSplash` — and #204 is also in `EnrichmentSplash`, so doing them together avoids re-touching the same component twice
- Header polish (#201) is independent

## Why **not** v1.0.0 release

Original framing of this milestone was "Wave 19 — Dog-food round 2 + v1.0.0 release." Emiel corrected: **v1.0.0 ships when every catalog plant has its illustration**. That's gated on the catalog illustration pipeline (#138, currently v1.1 backlog with the generation-source decision still open). #138 is therefore the real v1.0.0-blocker — not the polish items in this wave.

Wave 19 itself ships as a series of dog-food fixes; v1.0.0 tagging happens in a later wave once #138 has produced the 444 species illustrations.

## Risk + dependency notes

- **#202** is the highest-risk item: the investigation may surface a deeper schedule/calibration mismatch that's larger than this wave can absorb. If that happens, ship a hot-fix that makes the toast read from the same source as the chip (so they at least agree), and file a follow-up issue for any deeper calibration question.
- **#207 form validation** is the most-likely-to-grow item: validation conventions (required markers, error placement, focus order) tend to ripple. Cap this issue at the `/add` form. If patterns emerge that should generalize across all forms, file a follow-up — don't expand scope mid-wave.
- **#204 + #203** must sequence in that order (both touch `EnrichmentSplash`).
- All other issues are independent.

## Conventions

- **One PR per issue.** Public-repo etiquette: each PR closes exactly one issue with `Closes #NNN` in the body. Squash-merged via auto-merge per project default (per `feedback_plnt_trmnl_pr_flow.md`). The only legitimate exception is a genuinely atomic change that can't be split — that's not the case for any issue in this wave.
- **Push + auto-merge** the moment CI checks are armed (per `feedback_plnt_trmnl_pr_flow.md`).
- **Commit messages:** follow existing style — `fix(client): ...`, `feat(client): ...`, `chore(...): ...`. Reference the issue in the subject (`fix(client): #202 watering toast date matches displayed interval`).
- **Local rebuild:** `docker compose up -d --build` at the mid-wave checkpoint and at the wave-end gate; verify against feedback rows before flipping their status to `done`.
