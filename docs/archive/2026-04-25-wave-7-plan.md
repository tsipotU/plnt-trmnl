## Task 7: Diagnose the AddPlant flake

**Files:** none modified. Investigation only.

The failing test is `packages/api/client/src/pages/AddPlant.test.tsx:540`, inside the describe block `AddPlant — post-add enrichment splash (issue #72)`, test case `shows a splash with species + care preview when enrichment completes`. The assertion that fails is:

```tsx
await screen.findByRole('dialog', { name: /Sansevieria trifasciata/i });
```

The test submits the AddPlant form, which triggers polling of `/api/plants/7/enrichment-status`. The test mock uses a `statusCalls` counter (around line 494) and returns `enrichment_status: 'complete'` after a certain number of calls; the component then renders an `EnrichmentSplash` dialog whose accessible name includes the species.

- [ ] **Step 1: Confirm the flake rate on the current branch**

```bash
cd ~/Projects/plant-trmnl/packages/api/client
for i in 1 2 3 4 5; do
  echo "=== RUN $i ==="
  npx vitest run AddPlant.test.tsx 2>&1 | grep -E "Tests\s+[0-9]" | tail -1
done
```
Expected: 1-3 tests fail per run, unpredictably. You should see variation across runs. The most common failing test is the one at line 540.

- [ ] **Step 2: Read the component's polling logic**

Open `packages/api/client/src/pages/AddPlant.tsx`. Search for the string `enrichment-status` — this locates the polling `useEffect`. Note:
- The poll interval (look for `setInterval` or `setTimeout`).
- Whether the test uses `vi.useFakeTimers()` at the top of the `describe` block (check `AddPlant.test.tsx` lines ~486-495).
- Whether the test advances fake timers with `vi.advanceTimersByTime(...)` or relies on real timers.

- [ ] **Step 3: Form a hypothesis**

Most-likely causes, ranked:

1. **Default `findByRole` timeout too short** — `findByRole` defaults to 1000 ms. If the component's poll interval is 500 ms and the test uses real timers, one pass fetches status='pending', the second pass fetches 'complete', the dialog renders — all of which needs to fit inside 1000 ms including jsdom render overhead. On a loaded machine, this can miss.

2. **Missing `act()` around timer advance** — if the test uses fake timers but advances them without wrapping in `act()`, React's state updates may not flush before the query runs.

3. **Stale promise resolution order** — the `fetch` mock returns a resolved promise, but multiple polls can interleave and the component's state machine (pending → complete) may briefly land in an unexpected state.

- [ ] **Step 4: Capture diagnostic data**

Temporarily add a diagnostic `console.log` inside the existing fetch mock (around line 503–508) to print `statusCalls` and `status` on each call, AND add a `console.log(screen.debug())` just before the failing `findByRole`. Run the test 5× and note what `statusCalls` reached by the time the `findByRole` timed out. If `statusCalls` is less than what the mock needs to reach `complete`, the poll interval isn't firing enough. If `statusCalls` is correct but the DOM doesn't show the dialog, it's a render-flush issue.

**Remove the diagnostic logs before committing** (they're for your analysis only — don't commit them).

- [ ] **Step 5: Report the diagnosis**

Before moving to Task 8, write down (in your task report):
- The actual poll interval (from `AddPlant.tsx`).
- Whether the test uses fake or real timers.
- What `statusCalls` reached at timeout during your diagnostic runs.
- Which of the three hypotheses matches, or a new one.

Apply the **minimal** fix for the diagnosis you reached in Task 7. The three most likely fixes (pick one based on Task 7):

### Fix option A: extend findByRole timeout

If Task 7 showed `statusCalls` was correct but the render just hadn't completed in 1000 ms, extend the specific `findByRole` timeout:


### Fix option B: flush timers explicitly before the assertion

If the test uses `vi.useFakeTimers()` and `statusCalls` wasn't advancing, the test isn't flushing timers. Before the failing `findByRole` call, add:

```tsx
await vi.runAllTimersAsync();
```

If the diagnostic showed state-update warnings or flush issues, wrap timer advance explicitly:

```tsx
import { act } from '@testing-library/react';

await act(async () => {
```
